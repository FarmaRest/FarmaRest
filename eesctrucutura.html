¿Qué vamos a construir?

Una API REST de Productos con FastAPI, organizada en capas. Cada capa tiene una responsabilidad clara y separada.

💡 ¿Qué es la arquitectura de capas?

Es una forma de organizar el código en grupos con responsabilidades únicas. Cada capa solo habla con la capa inmediata de abajo. Esto hace el código más fácil de entender, probar y mantener.

Flujo de una petición HTTP
🌐

Recibe peticiones HTTP. Define los endpoints.

02

Domain Layer — Los Modelos

Define cómo lucen tus datos. Es la capa más fundamental: todas las demás la usan.

📐 domain/models.py
🤔 ¿Para qué sirve esta capa?

Define la forma y las reglas de tus datos. Usa Pydantic (incluido en FastAPI) para validar automáticamente que los datos sean correctos antes de procesarlos.

Archivo: domain/models.py
📁 domain / models.py Python

from pydantic import BaseModel
from typing import Optional


# ── Modelo base: campos comunes ──────────────────
class ProductBase(BaseModel):
    name: str
    description: str
    price: float
    stock: int


# ── Para CREAR un producto (sin ID) ─────────────
class ProductCreate(ProductBase):
    pass


# ── Respuesta completa (con ID) ──────────────────
class Product(ProductBase):
    id: int

    class Config:
        # Permite leer datos desde atributos de objeto
        from_attributes = True
L1 📦
Importamos BaseModel de Pydantic. Toda clase que herede de BaseModel valida automáticamente los datos que recibe.
L6–10 🧱
ProductBase define los campos comunes. Cada campo tiene su tipo de Python: str, float, int. Pydantic valida que vengan correctos.
L13–14 📥
ProductCreate hereda todo de Base. Se usa cuando el cliente envía datos para crear un producto — sin ID, porque el sistema lo genera.
L17–23 📤
Product agrega el campo id. Es el modelo que devuelve la API al cliente como respuesta. Config.from_attributes = True permite convertir objetos de base de datos a este modelo.
✅ Regla de esta capa

Solo define estructuras de datos. No tiene lógica, no accede a la base de datos. Es el vocabulario que hablan todas las capas.

03

Repository Layer — Acceso a datos

Esta capa es la única que sabe cómo guardar y leer datos. Si mañana cambias de base de datos, solo tocas esta capa.

🗄️ repository/product_repo.py
💡 Patrón Repository

Esconde cómo se guardan los datos. El Service no sabe si los datos vienen de PostgreSQL, MongoDB, un archivo JSON, o como en este caso, de un diccionario en memoria. Eso es encapsulamiento.

Archivo: repository/product_repo.py
📁 repository / product_repo.py Python
12345 678910 1112131415 1617181920 2122232425 2627282930 3132333435 3637383940 
from typing import Optional, List
from domain.models import Product, ProductCreate


class ProductRepository:
    """Simulamos una base de datos con un dict."""

    def __init__(self):
        # "Base de datos" en memoria
        self._db: dict[int, Product] = {}
        self._next_id: int = 1

    def get_all(self) -> List[Product]:
        # Devuelve todos los productos
        return list(self._db.values())

    def get_by_id(self, product_id: int) -> Optional[Product]:
        # Busca por ID. None si no existe
        return self._db.get(product_id)

    def create(self, data: ProductCreate) -> Product:
        # Asigna ID, guarda y retorna el producto
        product = Product(
            id=self._next_id,
            **data.model_dump()
        )
        self._db[self._next_id] = product
        self._next_id += 1
        return product

    def delete(self, product_id: int) -> bool:
        # True si existía y se borró
        if product_id in self._db:
            del self._db[product_id]
            return True
        return False
L2 🔗
Importa los modelos del Domain Layer. El Repository usa esos modelos para hablar con el resto del sistema.
L8–11 🗃️
self._db es un diccionario que actúa como base de datos. La clave es el id y el valor es el objeto Product. En un proyecto real, aquí iría SQLAlchemy o un ORM.
L13–15 📋
get_all() devuelve todos los valores del diccionario como lista. List[Product] en el type hint le dice a Python (y a FastAPI) qué tipo de dato esperar.
L22–28 ✏️
data.model_dump() convierte el objeto Pydantic a un dict. El ** lo "desempaca" como argumentos con nombre. Es la forma estándar de crear un modelo desde otro.
L30–35 🗑️
delete() retorna True o False para indicar si la operación tuvo éxito. El Service usa ese resultado para decidir qué responder al cliente.
04

Service Layer — La lógica de negocio

Aquí vive el cerebro de tu aplicación. Orquesta operaciones, aplica reglas de negocio y decide qué puede o no puede hacerse.

⚙️ service/product_service.py
🧠 ¿Qué va aquí?
  • Validaciones de negocio (ej: el precio no puede ser negativo)
  • Reglas (ej: no se puede crear un producto sin stock)
  • Orquestación (llama al Repository para leer/escribir datos)
  • Manejo de errores específicos del negocio
Archivo: service/product_service.py
📁 service / product_service.py Python
12345 678910 1112131415 1617181920 2122232425 2627282930 3132333435 3637383940 4142434445 
from typing import List
from fastapi import HTTPException, status

from domain.models import Product, ProductCreate
from repository.product_repo import ProductRepository


class ProductService:

    def __init__(self, repo: ProductRepository):
        # Recibe el repositorio por inyección de dependencias
        self.repo = repo

    def get_all_products(self) -> List[Product]:
        return self.repo.get_all()

    def get_product(self, product_id: int) -> Product:
        product = self.repo.get_by_id(product_id)

        # ⬇ Regla de negocio: si no existe, lanza error 404
        if not product:
            raise HTTPException(
                status_code=status.HTTP_404_NOT_FOUND,
                detail=f"Producto {product_id} no encontrado"
            )
        return product

    def create_product(self, data: ProductCreate) -> Product:
        # ⬇ Regla de negocio: precio no puede ser negativo
        if data.price < 0:
            raise HTTPException(
                status_code=status.HTTP_400_BAD_REQUEST,
                detail="El precio no puede ser negativo"
            )
        return self.repo.create(data)

    def delete_product(self, product_id: int) -> dict:
        deleted = self.repo.delete(product_id)
        if not deleted:
            raise HTTPException(
                status_code=status.HTTP_404_NOT_FOUND,
                detail=f"Producto {product_id} no encontrado"
            )
        return {"message": "Producto eliminado correctamente"}
L2 🚨
HTTPException es la forma estándar de FastAPI para lanzar errores HTTP con un código de estado (404, 400, etc.) y un mensaje claro para el cliente.
L10–12 💉
El __init__ recibe el repositorio como parámetro. Esto se llama Inyección de Dependencias. El Service no crea el repositorio, solo lo usa. Facilita testear con repositorios falsos.
L21–25 🔴
Si el producto no existe, lanzamos HTTP 404. FastAPI convierte automáticamente esta excepción en una respuesta JSON con el código y el mensaje de error.
L29–33 📏
Aquí está la lógica de negocio real: validar que el precio sea positivo antes de guardar. El Repository no sabe de estas reglas — ese es el trabajo del Service.
05

API Layer — Los Endpoints

La "puerta de entrada" de tu API. Recibe las peticiones HTTP, llama al Service y devuelve la respuesta.

🌐 api/router.py
🌐 ¿Qué hace esta capa?
  • Define las rutas: GET /products, POST /products, etc.
  • Extrae los datos de la petición (body, path params, query params)
  • Delega el trabajo al Service (no tiene lógica propia)
  • Define el código HTTP de respuesta correcto
Archivo: api/router.py
📁 api / router.py Python
12345 678910 1112131415 1617181920 2122232425 2627282930 3132333435 3637383940 4142434445 4647484950 
from typing import List
from fastapi import APIRouter, status

from domain.models import Product, ProductCreate
from service.product_service import ProductService
from repository.product_repo import ProductRepository


# Instancias (en producción usarías DI de FastAPI)
repo = ProductRepository()
service = ProductService(repo)

# Router: agrupa endpoints bajo un prefijo
router = APIRouter(prefix="/products", tags=["products"])


@router.get("/", response_model=List[Product])
def list_products():
    """Devuelve todos los productos."""
    return service.get_all_products()


@router.get("/{product_id}", response_model=Product)
def get_product(product_id: int):
    """Devuelve un producto por su ID."""
    return service.get_product(product_id)


@router.post(
    "/",
    response_model=Product,
    status_code=status.HTTP_201_CREATED
)
def create_product(data: ProductCreate):
    """Crea un nuevo producto."""
    return service.create_product(data)


@router.delete(
    "/{product_id}",
    status_code=status.HTTP_200_OK
)
def delete_product(product_id: int):
    """Elimina un producto por su ID."""
    return service.delete_product(product_id)
L13 🗂️
APIRouter agrupa endpoints. El prefix="/products" hace que todos los endpoints comiencen con /products. El tags organiza la documentación automática de Swagger.
L16 🎯
@router.get("/") es un decorador. Le dice a FastAPI que cuando llegue una petición GET a /products/, ejecute la función de abajo. response_model define cómo se serializa la respuesta.
L23 🔍
{product_id} en la ruta es un path parameter. FastAPI lo extrae de la URL y lo pasa como argumento a la función — automáticamente convertido a int.
L29–35 📮
status_code=HTTP_201_CREATED devuelve 201 en vez de 200 al crear. El parámetro data: ProductCreate indica a FastAPI que espere un JSON body y lo valide automáticamente.

main.py — El punto de entrada

Aquí se ensambla toda la aplicación. Crea la instancia de FastAPI y registra los routers.

from fastapi import FastAPI
from api.router import router


# Crea la aplicación FastAPI
app = FastAPI(
    title="Mi API de Productos",
    description="API REST con arquitectura de capas",
    version="1.0.0"
)

# Registra el router de productos

# Endpoint raíz de bienvenida
@app.get("/")
def root():
    return {"message": "¡API funcionando! 🚀"}
L5–9 🏗️
FastAPI(...) crea la aplicación. Los parámetros title, description y version aparecen automáticamente en la documentación interactiva (Swagger UI).
L12 🔌
include_router(router) conecta todos los endpoints definidos en api/router.py a la aplicación. Si tuvieras más recursos (usuarios, pedidos) solo agregarías más routers aquí.

🎉 ¿Qué hace --reload?

Le dice a Uvicorn que reinicie automáticamente el servidor cada vez que guardes un archivo. Ideal para desarrollo. En producción NO se usa.

📖 main:app ¿qué significa?

main es el nombre del archivo Python (main.py) y app es la variable dentro de ese archivo donde creaste la instancia de FastAPI. Uvicorn necesita saber ambas cosas.
